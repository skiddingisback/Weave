Introduction › About

Weave

A typed, diagnostics-first dependency-injection framework for Roblox — a from-first-principles redesign of the space Knit occupied.

Weave is built for large production games: hundreds of modules, dozens of services, multiple developers, Rojo, CI, automated tests, and hot reload. It is written in strict Luau and has zero runtime dependencies.

Why Weave?

  • Typed dependency injection. Services are looked up by typed tokens. You get full autocomplete on every dependency, everywhere, and implementations never require each other — so large graphs stay cycle-free at the module level.
  • One abstraction for both sides. Servers and clients are both described with a single Weave.service primitive. A "service" runs on the server; a "controller" is simply a service that runs on the client.
  • Deterministic startup. Declared dependencies drive a topological boot order, cycle detection, and fail-fast lifecycle phases. Shutdown runs in reverse.
  • Schema-first networking. RPCs and events are validated on both sides against a shared schema, with rate limiting, middleware, and a version fingerprint that catches deploy skew.
  • Diagnostics you can read. Startup timelines, dependency graphs (Mermaid & DOT), and structured errors that suggest concrete fixes.
Coming from Knit?

The mental model is familiar: services, controllers, lifecycle, networking. The big differences are typed tokens instead of string keys, declared dependencies with an explicit boot order, and validated networking. See Getting Started.

Introduction › Getting Started

Getting Started

Install Weave, lay out your project, and boot your first app.

Installation

Wally (recommended)

[dependencies]
Weave = "skiddingisback/weave@0.1.0"

Then run wally install and require it from your Packages folder:

local Weave = require(game.ReplicatedStorage.Packages.Weave)

Rojo (manual)

Drop the library's src folder anywhere shared and point Rojo at it:

{ "ReplicatedStorage": { "Weave": { "$path": "path/to/Weave/src" } } }

Project layout

A conventional structure keeps each service in its own folder, with a definition module (the typed contract) beside its implementation:

ReplicatedStorage/
  Weave/                 -- the framework
  Shared/
    DataService/
      definition.luau    -- token + API type (required by everyone)
ServerScriptService/
  Services/
    DataService/
      init.luau          -- implementation (Weave.service{...})
  Registry.luau          -- lists every service to register

Bootstrap the server

local Weave = require(game.ReplicatedStorage.Weave)

local app = Weave.App.new({ name = "Server" })
app:registerMany(require(game.ServerScriptService.Registry))

local report = app:start()
print(report.timelineText)   -- startup timeline
app:bindToClose()            -- graceful shutdown

Bootstrap the client

local Weave = require(game.ReplicatedStorage.Weave)

local app = Weave.App.new({ name = "Client" })
app:registerMany(require(game.ReplicatedStorage.Client.Registry))
app:start()
Tip

A Registry module just returns an array of Weave.service{...} results. Keeping registration in one place makes the boot order and dependency graph easy to reason about.

Core › Services

Services

A service is a long-lived singleton with a typed API and declared dependencies. Services run on the server.

1. The definition module

Every service has a small definition module that exports its API type and a token cast against it. This is the module both implementations and consumers require — it carries no logic, so requiring it never creates a cycle.

-- DataService/definition.luau
local Weave = require(game.ReplicatedStorage.Weave)

export type API = {
    read: (self: API, player: Player, key: string) -> number,
    write: (self: API, player: Player, key: string, value: number) -> (),
}

return { Token = Weave.token("DataService") :: Weave.Token<API> }

2. The implementation

The implementation returns Weave.service{...}. Dependencies are declared (that is what makes ordering and cycle detection possible) and resolved with full types via ctx:use.

-- ShopService/init.luau
local Weave = require(game.ReplicatedStorage.Weave)
local Def = require(script.definition)
local DataDef = require(script.Parent.DataService.definition)

return Weave.service({
    token = Def.Token,
    dependencies = { DataDef.Token },
    create = function(ctx): Def.API
        local data = ctx:use(DataDef.Token)   -- typed as DataDef.API
        local self = {} :: Def.API

        function self.buy(_, player, itemId)
            local balance = data:read(player, "balance")
            -- ...
        end

        ctx:onStart(function()
            ctx.log:info("shop open")
        end)

        return self
    end,
})

The service context

create receives a ctx with everything the service needs:

MemberDescription
ctx:use(token)Resolve a declared dependency (typed).
ctx:useOptional(token)Resolve if present, else nil.
ctx.logA scoped logger.
ctx:own(item)Track a resource; released at shutdown.
ctx:spawn(fn)Managed loop/thread, cancelled at shutdown.
ctx:onStart(fn)Fail-fast startup hook, runs in topo order.
ctx:onReady(fn)Runs after all services started (errors non-fatal).
ctx:onShutdown(fn) / ctx:onDestroy(fn)Teardown hooks, run in reverse boot order.
ctx:getConfig(key) / ctx:getMetadata(key)Read app config / service metadata.
Don't loop in onStart

onStart is fail-fast and sequential. Long-running loops belong in ctx:spawn(...), not in an onStart body, or startup will block.

Registering & starting

local app = Weave.App.new({ name = "Server" })
app:register(require(ShopService))         -- one
app:registerMany(require(Registry))         -- or many
app:provide(ConfigToken, { maxItems = 20 }) -- plain values
app:start()
Core › Controllers

Controllers

A controller is a client-side service. It uses the exact same Weave.service primitive — there is no separate API to learn.

On the client you create a client App, register your controllers, and start it. Controllers typically own input, UI, and a NetClient that talks to server services.

-- ShopController/init.luau
local Weave = require(game.ReplicatedStorage.Weave)
local Def = require(script.definition)
local NetDef = require(script.Parent.Net.definition)

return Weave.service({
    token = Def.Token,
    dependencies = { NetDef.Token },
    create = function(ctx): Def.API
        local net = ctx:use(NetDef.Token)
        local self = {} :: Def.API

        function self.buy(_, itemId)
            local ok, res = net:call(ShopApi.buy, { itemId = itemId })
            return ok and res or nil
        end

        ctx:onStart(function()
            ctx.log:info("shop UI ready")
        end)

        return self
    end,
})
Server vs. client

The only difference between a service and a controller is which App registers it. Weave will error clearly if a server-only dependency is resolved from a client App (or vice versa).

Client bootstrap

local app = Weave.App.new({ name = "Client" })
app:registerMany(require(game.ReplicatedStorage.Client.Registry))
app:start()

app:onReady(function()
    print("all controllers started")
end)
Core › Util

Util

Weave ships a small set of dependency-free utilities used throughout the framework — and usable on their own.

ModulePurpose
Weave.SignalTyped event emitter (Connect, Once, Wait, Fire, Destroy).
Weave.FutureA settle-once async value (await, andThen, cancel).
Weave.BinCleanup collector for connections, instances, threads, and functions.
Weave.LogLeveled, scoped logging with a swappable sink.
Weave.ErrorStructured errors with a code, chain, and suggested fixes.
Weave.CheckRuntime validators (used by the network schema).
Weave.Timeline / Weave.GraphExportStartup diagnostics and graph export.

Signal

local bought = Weave.Signal.new() :: Weave.Signal<Player, string>
local conn = bought:Connect(function(player, itemId) end)
bought:Fire(player, "sword")
conn:Disconnect()

Bin

local bin = Weave.Bin.new()
bin:own(part.Touched:Connect(onTouch))  -- RBXScriptConnection
bin:own(model)                           -- Instance
bin:own(function() print("cleanup") end) -- function
bin:destroy()                            -- releases everything, in reverse

Future

local ok, value = someFuture:await(5)    -- yields up to 5s
someFuture:andThen(function(ok, value) end)
Tip

Inside a service, prefer ctx:own(...) over a manual Bin — the framework destroys it for you when the service shuts down.

Core › Execution Model

Execution Model

Exactly what happens between app:start() and a running game — and how it tears down.

Startup phases

  1. configure — each service's optional configure(ctx) runs. It can disable() itself (feature flags / platform gating) or set metadata.
  2. resolve — declared dependencies are checked and topologically sorted. Missing or cyclic dependencies fail here with a readable error.
  3. init — services are constructed in dependency order (each create runs once). A watchdog warns if a step yields too long.
  4. start — onStart callbacks run, fail-fast, in boot order. Any error unwinds already-started services in reverse.
  5. ready — onReady callbacks run. Errors here are reported as warnings, not fatal.
Fail-fast vs. best-effort

init and start are fail-fast: a thrown error aborts startup and cleanly unwinds. ready is best-effort: a failing onReady is logged and collected in the start report's warnings.

Ordering & cycles

The boot order is a topological sort of the declared dependency graph, tie-broken by registration order for stable output. A cycle produces E_CIRCULAR_DEPENDENCY with the exact chain. Break cycles by extracting a shared service, or make one edge optionalDependencies and resolve it lazily.

Lazy services

Mark a service lazy = true to register and cycle-check it without constructing it at boot. It builds on first app:activate(token) or first resolve:

return Weave.service({
    token = Def.Token,
    lazy = true,
    create = function(ctx) ... end,
})

-- later, on demand:
local feature = app:activate(Def.Token)

Shutdown

app:shutdown() (and app:bindToClose() on the server) runs onShutdown then onDestroy in reverse boot order, destroys every service's Bin, and disposes the container.

The start report

local report = app:start()
print(report.timelineText)   -- per-service timing table
print(report.graphMermaid)   -- paste into a Markdown/PR
print(report.warnings)       -- non-fatal issues
Guides › Examples

Examples

End-to-end snippets that put the pieces together.

A service with a dependency

-- DataService/definition.luau
export type API = { read: (self: API, p: Player, k: string) -> number }
return { Token = Weave.token("DataService") :: Weave.Token<API> }

-- ShopService/init.luau
return Weave.service({
    token = ShopDef.Token,
    dependencies = { DataDef.Token },
    create = function(ctx): ShopDef.API
        local data = ctx:use(DataDef.Token)
        local self = {} :: ShopDef.API
        function self.getBalance(_, player)
            return data:read(player, "balance")
        end
        return self
    end,
})

A validated RPC

-- shared schema
local Check = Weave.Check
local ShopApi = Weave.Net.namespace("Shop", {
    buy = Weave.Net.rpc(
        Check.shape({ itemId = Check.string }),      -- request
        Check.shape({ ok = Check.boolean }),          -- response
        { rate = 10 }                                 -- 10 req/s per player
    ),
})

-- server
net:handle(ShopApi.buy, function(player, req)
    return { ok = shop:buy(player, req.itemId) }
end)

-- client
local ok, res = net:call(ShopApi.buy, { itemId = "sword" })

Broadcasting an event

local Events = Weave.Net.namespace("Shop", {
    priceChanged = Weave.Net.event(Check.shape({ itemId = Check.string, price = Check.number })),
})

-- server
net:broadcast(Events.priceChanged, { itemId = "sword", price = 120 })

-- client
net:on(Events.priceChanged, function(payload)
    updatePriceTag(payload.itemId, payload.price)
end)
Guides › Middleware

Middleware

Cross-cutting logic for networking — auth, logging, throttling — composed as a pipeline.

Server middleware

Each middleware receives a ServerNetContext and a next function. Call next() to continue the chain; set a deny code to reject the request before it reaches its handler.

local function requireAuth(ctx, next)
    if not isAuthed(ctx.player) then
        ctx.denyCode = "E_UNAUTHORIZED"
        ctx.denyMessage = "sign in first"
        return            -- do NOT call next(): request is denied
    end
    next()
end

local net = Weave.Net.Server.new({
    transport = Weave.Net.RobloxTransport.createServer({ fingerprint = fp }),
    namespaces = { ShopApi },
    middleware = { requireAuth },
})

The context

FieldDescription
ctx.playerThe sending player (peer).
ctx.endpointId / ctx.kindWhich endpoint, and "rpc" or "event".
ctx.payloadThe raw (pre-validation) payload.
ctx.metaScratch table to pass data down the chain.
ctx.denyCode / ctx.denyMessageSet to reject the request.

Rate limiting

Every endpoint is rate limited with a token bucket. Set rate (per second) and optional burst per endpoint, or defaults on the server:

buy = Weave.Net.rpc(reqCheck, resCheck, { rate = 10, burst = 20 })

Outbound interceptors

Transform or observe every outgoing payload (e.g. compression, analytics) via outbound on the server/client options.

Generic pipelines

Weave.Pipeline.compile(steps, terminal) builds the same onion-style chain for any context type, not just networking.

Guides › Intellisense

Intellisense

Weave is designed so your editor knows the type of every service, dependency, and network payload.

How it works

The definition.luau pattern is the key. Because a token is cast against an exported API type:

return { Token = Weave.token("DataService") :: Weave.Token<API> }

…every ctx:use(DataDef.Token) resolves to DataDef.API, so method names, arguments, and return types all autocomplete — across module and even actor boundaries.

Editor setup

Weave is a Rojo project, so luau-lsp can type-check against a sourcemap:

rojo sourcemap default.project.json --output sourcemap.json
luau-lsp analyze --sourcemap=sourcemap.json src

In VS Code, the luau-lsp extension auto-generates the sourcemap. The repo ships a .vscode/settings.json and a .luaurc that enable strict mode and Roblox types.

Definition modules carry no logic

Keeping the token + types in a separate, logic-free module is what lets everyone require it without creating a dependency cycle at the require level.

Scaffold Weave definitions and services in a couple of keystrokes.

Create .vscode/weave.code-snippets in your project and paste the following. Type the prefix (e.g. wservice) and press Tab.

{
  "Weave definition module": {
    "prefix": "wdef",
    "body": [
      "local Weave = require(game.ReplicatedStorage.Weave)",
      "",
      "export type API = {",
      "\t$0",
      "}",
      "",
      "return { Token = Weave.token(\"${1:ServiceName}\") :: Weave.Token }"
    ],
    "description": "Weave token + API definition module"
  },
  "Weave service": {
    "prefix": "wservice",
    "body": [
      "local Weave = require(game.ReplicatedStorage.Weave)",
      "local Def = require(script.definition)",
      "",
      "return Weave.service({",
      "\ttoken = Def.Token,",
      "\tdependencies = { $2 },",
      "\tcreate = function(ctx): Def.API",
      "\t\tlocal self = {} :: Def.API",
      "\t\t$0",
      "\t\tctx:onStart(function() end)",
      "\t\treturn self",
      "\tend,",
      "})"
    ],
    "description": "Weave service implementation"
  },
  "Weave RPC": {
    "prefix": "wrpc",
    "body": [
      "${1:name} = Weave.Net.rpc(",
      "\tWeave.Check.shape({ $2 }),",
      "\tWeave.Check.shape({ $3 })",

    ],
    "description": "Weave networked RPC endpoint"

  }
}
